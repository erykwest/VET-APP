# UX Review Log - 2026-03-26

This note tracks the first aesthetic review performed on the Flutter web auth/onboarding flow and the quick fixes applied.

## Context

Observed from the current browser preview:

- buttons, inputs, banners, and sections stretched across the full available width on desktop
- hero panel, state banner, and form had similar visual weight
- desktop layout felt too wide for form reading and conversion-focused tasks

## Quick fix applied

The auth and onboarding shells now center content and cap the main column width to improve readability on desktop while preserving mobile behavior.

Affected areas:

- auth scaffold
- onboarding scaffold
- onboarding welcome page

## Why this change matters

On desktop, full-width forms reduce scan speed and make CTA grouping feel less intentional.

A bounded content column improves:

- readability
- visual rhythm
- perceived product polish
- separation between hero, status, and form sections

## UX findings after the first pass

### Strong points

- warm-clinical palette is coherent with the product direction
- rounded surfaces and soft gradients already give the app a recognizable tone
- copy is directional and product-like, not generic placeholder text

### Issues still visible

- primary and secondary CTAs are still duplicated between hero and form areas
- the status banner can compete too much with the actual form content
- several desktop sections still rely on full-width internal controls even when the page shell is constrained
- typographic hierarchy is improved but still a bit flat in some forms

## Suggested next UX pass

1. Reduce CTA duplication between hero and form.
2. Rebalance the status banner so it supports the task instead of dominating it.
3. Introduce desktop-specific max widths for inner form groups where needed.
4. Refine spacing and label contrast for a more premium reading rhythm.
5. Review cross-feature consistency for pets, records, reminders, and profile pages.

## Validation status

- targeted analysis on the touched UX files passed after cleanup
- full-package analysis was not treated as the acceptance gate because the worktree contains unrelated local changes
